# Northline Retail Order Fulfilment Delay Business Case

Northline Retail is a fictional e-commerce retailer used for this project, operating five regional warehouses with an order history of roughly 11,000 records. Payment and stock availability are confirmed at checkout, so an order only enters fulfilment once both are already resolved.

When a customer places an order, the system allocates it to one of Northline's five warehouses, following standard multi-warehouse logic: stock availability first, then proximity to the delivery address, then load balancing if more than one warehouse qualifies. From there, the order goes to a supervisor for approval, which is currently applied to every single order regardless of its size, value, or risk. Once approved, warehouse staff pick and pack it. If something goes wrong during picking, a stock discrepancy or a misplaced item, staff log it and escalate to the supervisor, who resolves it or places the order on hold. Once packed, the order is handed to a carrier and delivered.

At no point in this sequence does anyone check whether this specific order, based on its weight, its discount, or which warehouse is handling it, is more likely to be late than any other. Every order moves through the same steps at the same pace, and the only signal Northline has that something went wrong is the delivery itself arriving late. By then, the only response left is reactive, usually a customer support complaint that ends in a refund or a discount code to avoid a bad review. A late delivery also makes a repeat order less likely, since delivery timing is one of the few parts of the experience a customer feels directly.

This project uses the Kaggle E-Commerce Shipping dataset (kaggle.com/datasets/prachi13/customer-analytics), roughly 11,000 historical orders, as a stand-in for Northline's own order history. The dataset includes order weight, discount level, warehouse block, shipment mode, and whether each order arrived on time, enough to test which of these, if any, actually predicts delay, rather than assuming an answer up front. That analysis, and the process change it points to, are covered in the project's SQL, dashboard, and process diagrams.

This project sets out to identify the factors behind delayed deliveries through business and data analysis, then recommends a process change that flags at-risk orders automatically before dispatch, instead of the business finding out only after the customer has already complained.

## Limitations and Next Steps

Northline's name, scale, and warehouse operations are invented for this project and have no connection to the dataset's original source. The dataset itself has no timestamps, so it isn't possible to check whether delay patterns change over time or by season. Any threshold this project settles on is based on a fixed historical snapshot and would need to be retested if Northline's discount policy changes. A natural next step would be a correlation or feature selection analysis using timestamped fulfilment data, order confirmation, pick, pack, and dispatch times, to see where in the process delay actually accumulates beyond what this project's confirmed risk factors explain.
